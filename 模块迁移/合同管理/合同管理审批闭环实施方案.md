# 合同管理审批闭环实施方案

> 当前版本只写实施方案，不改流程节点配置。合同审批、退租审批、房屋验收、逾期律师函审批、付款/开票流程审批的流程已由协同办公/工作流模块配置，本方案只负责把合同管理业务页面、内置审批表单、审批状态回写、原样打印模板和合同后续动作串起来。

## 1. 目标

合同管理需要从“创建合同”升级为完整闭环：

1. 管理后台创建合同，选择合同模板，填入客户、房源、租期、租金等信息。
2. 发起合同审批，内置审批表单承载审批内容，审批通过后生成合同会签审批表、合同正文等文件。
3. 上传盖章合同，合同生效，并自动生成合同付款账单。
4. 付款账单进入费用缴纳、财务确认、付款通知/开票/押金退还等流程。
5. 逾期未缴费进入催缴、逾期处理通知、律师函审批、退租/解除合同等后续流程。
6. 退租、续租都能从合同档案继续发起，合同变更以补充协议上传归档方式沉淀附件和结果。

所有审批表、通知书、合同、补充协议、交接单必须按原模板打印，目标是“版式一模一样”，不能用普通 HTML 表格临摹代替最终打印。

## 2. 不做范围

本阶段不处理流程节点配置。节点、候选人、会签、抄送、条件流、按钮等继续在工作流设计器里维护。

本阶段不继续扩展旧的 `biz_approval_*` 自研审批链路。新审批统一走 NutFlow/Flowable，当前合同相关流程优先使用工作流设计器配置的内置表单。

本阶段不写死具体流程 key。合同管理页面只按业务类型筛选/默认选择已部署流程，用户可以在弹窗里切换流程。流程 key、表单 key 的当前配置见下方“当前流程与表单 key 清单”。

## 3. 当前可复用基础

### 3.1 入驻审批模式

入驻审批已经形成可复用模式：

- 前端入口：`saber3/src/views/settlement/project-approval.vue`
- 外部表单：`saber3/src/views/plugin/workflow/pages/external/TenantEntry/start.vue`
- 外部详情：`saber3/src/views/plugin/workflow/pages/external/TenantEntry/detail.vue`
- 后端流程事件：`TenantEntryWorkflowServiceImpl`
- 工作流通知分发：`WfNoticeServiceImpl`
- 发起方式：外部表单通过 `startProcessByKey` 发起流程，并把 `businessType`、`businessKey`、业务 ID 写入流程变量。

合同管理可以复用这个模式里的“发起流程、状态回写、审批详情、打印导出”能力，但合同相关流程当前已经配置为内置表单，后续开发优先接入内置表单，不再优先新增合同外部表单页面。

### 3.2 合同模块现状

当前合同模块已有基础数据和账单能力：

- 合同主档：`biz_contract`
- 合同付款计划：`biz_contract_payment`
- 合同日志：`biz_contract_log`
- 创建合同时会根据租期、月租金、物业费、管理费、公摊费、付款周期生成付款计划。
- 已入驻客户校验已经存在，合同创建可关联客户、园区、楼宇、房源。

## 5. 流程图拆解

### 5.1 主线：创建合同到盖章归档

1. 管理后台创建合同。
2. 选择合同模板：固定租金/浮动租金/后续上传模板。
3. 自动带出已入驻企业、楼宇、房间、面积、租期、租金、押金等数据。
4. 生成合同草稿。
5. 发起合同审批。
6. 审批内容包含：
   - 合同起草，招商部负责。
   - 合同文本审核，风控部负责。
   - 内部审批，使用合同会签审批表、用印申请单等材料。
7. 审批通过后生成正式审批文件。
8. 上传盖章合同。
9. 合同状态变为已生效。
10. 自动生成合同应付账单。

### 5.2 付款主线

1. 合同生效后生成应付账单。
2. 账单进入费用缴纳。
3. 财务确认缴费。
4. 根据业务需要生成付款通知单、开票申请、收款/付款审批记录。
5. 付款通过后回写账单状态。

### 5.3 逾期主线

1. 账单到期未缴费，进入逾期缴费列表。
2. 一旦逾期，系统提醒招商员、分管副总、部门领导、财务。
3. 逾期 1 个月仍未缴纳，租金逾期处理通知书通知招商员、分管副总、部门领导。
4. 仍未缴纳，发起项目审批/逾期律师函审批。
5. 审批通过后生成催款通知书、租金逾期处理通知书或律师函相关材料。
6. 拒不返还或仍不处理，进入限期搬离通知/退租流程。

### 5.4 退租主线

1. 正常退租：租赁期限内提前 30 日提醒招商员。
2. 发起退租审批，内置表单使用退租审批表。
3. 审批附件包含租赁合同、复印件等材料。
4. 审批通过后生成合同解除补充协议。
5. 线下盖章后上传归档。
6. 归还承租主体，上传费用支付凭证、现场验收情况等附件。
7. 发起押金退还/付款审批。
8. 退租完成后释放房源，合同状态变为已终止。

当前退租流程图落地口径：

1. 合同详情发起退租申请时，先选择退租类型：正常退租、客户提前退租、特殊情况。
2. 正常退租默认退租日为发起日后 30 天，走 `termination` 退租审批，审批材料包含退租申请、租赁合同复印件等附件；审批通过后自动生成《退租审批表》和《租赁合同解除补充协议》。
3. 提前退租分两类处理：客户自行提前退租时必须记录违约金并进入退租审批；特殊情况标记为线下一事一议，但仍保留线上流程记录和归档入口。
4. 退租审批发起时带入 `terminationType`、`terminationTypeName`、`expectedTerminationDate`、`breachPenalty`、`terminationReason`、`offlineSpecialProcess` 等流程变量；审批通过后后端使用这些变量回填原始 Word 模板。
5. 解除补充协议线下盖章后作为附件归档；系统先提供自动生成文件地址，后续可继续补上传盖章件字段。
6. 退租审批通过后合同状态进入“退租待交接”，退租管理页显示“房屋验收”入口。
7. 房屋验收走 `roomreview` 流程，支持上传费用支付凭证、现场验收情况、载体验收附件；审批通过后合同状态变为“已终止”，并释放房源为空置。
8. 房屋验收完成后，退租管理页显示“押金退还”入口；系统按合同保证金生成一条 `deposit_refund` 付款单，再走 `pay` 付款审批，审批通过后回写付款单状态。

### 5.5 续租与补充协议归档

续租从合同管理发起，不单独走审批流程；系统带出原合同客户、房源、租金、押金等数据新建续租合同，原合同标记为已续签，新合同后续按普通合同主线处理。

合同变更以线下流程为主，不再设置“合同变更审批”二级菜单；业务人员在合同归档中上传补充协议完成线上归档。后续如果需要审批，可以按同样工作流模式扩展。

当前流程图拆成以下开发口径，明天开发按这张表验收是否串起来：

| 流程图节点 | 系统动作 | 对应流程/表单 | 开发重点 |
| --- | --- | --- | --- |
| 管理后台创建合同 | 创建合同草稿，选择合同模板，带出客户、房源、租期、租金 | 不发起流程 | 合同基础信息、合同模板、合同草稿文件 |

| 退租申请 | 合同详情发起退租审批 | `termination` / 退租审批表 | 退租审批通过后生成解除补充协议 |
| 归还载体/房屋验收 | 退租审批后发起房屋验收 | `roomreview` / 房屋交接验收单 | 上传费用支付凭证、验收附件、验收结果 |

| 合同变更 | 线下流程，合同归档内上传补充协议 | 暂不发起流程 | 补充协议上传、合同档案归档 |

## 7. 数据模型建议

审批类型多，而且同一合同可能多次发起退租、付款、逾期处理，所以不建议只在 `biz_contract` 上堆很多流程字段。建议新增一张通用合同流程记录表。

### 7.1 `biz_contract_workflow_record`

核心字段建议：

| 字段 | 说明 |
| --- | --- |

## 8. 表单设计与运行方式

当前合同相关流程优先使用工作流设计器中的内置表单。内置表单负责审批过程中的字段采集、默认值、远端字典和附件上传；打印文件由后端根据原始 Word/Excel 模板生成。

合同管理发起审批时走内置表单路由：

```text
/plugin/workflow/pages/process/form/start/{base64参数}
```

参数示例：

```json
{
  "processDefKey": "contract_sign",
  "params": {
    "businessType": "contract_approval",
    "contractId": "合同ID",
    "paymentId": "付款计划ID",
    "roomReviewId": "房屋验收ID"
  }
}
```

内置表单启动页会根据 `processDefKey` 读取当前部署表单，并把 `params` 作为表单默认值来源。合同业务页面负责在跳转前准备业务 ID、业务类型和必要默认值；后端流程事件负责根据流程变量回写业务状态。

如果某类审批后续需要更强的业务交互能力，再补外部表单。外部表单不是明天第一阶段开发重点。

## 10. 打印模板方案

为了保证“打印出来格式一模一样”，建议使用原始 Word/Excel 模板填充，不采用手写 HTML 表格作为正式输出。

### 10.1 模板处理原则

1. 原模板作为母版保存，不能直接覆盖。
2. 旧格式 `.doc`、`.xls` 先转换为 `.docx`、`.xlsx` 作为可填充模板。
3. 每份模板建立字段映射，例如 `${customerName}`、`${contractNo}`、`${rentArea}`。
4. 后端复制模板，填充字段，生成业务文件。
5. 生成后的 Word/Excel 再转换为 PDF。
6. 文件地址回写到合同流程记录、合同主档或付款计划。

### 10.2 后端组件建议

建议新增：

- `ContractTemplateService`：模板登记、字段映射、模板版本管理。
- `ContractPrintService`：按模板 key + 业务数据生成 Word/Excel/PDF。
- `ContractWorkflowService`：审批事件回写、结束时调用打印服务。
- `ContractWorkflowRecordService`：统一保存流程记录。

### 10.3 打印质量验收

每个模板至少做一次视觉验收：

1. 原始模板导出 PDF。
2. 填充后的模板导出 PDF。
3. 对比页数、边距、表格线、字体、换行、签字栏位置。
4. 字段内容过长时，优先自动换行，不能撑破表格。
5. 审批签字/意见区域保留原模板空间，不在打印阶段重排。

## 13. 各流程通过后的业务动作

### 13.1 合同审批通过

1. 合同状态：审批通过待盖章。
2. 生成合同会签审批表。
3. 生成合同正文。
4. 合同列表显示“上传盖章合同”。
5. 上传盖章合同后，合同状态变为已生效。
6. 自动生成付款账单。

### 13.2 付款流程通过

1. 付款计划状态更新。
2. 已付款金额、付款时间、付款凭证回写。
3. 生成付款通知单/开票申请单。
4. 如果是押金退还，回写退租记录。

### 13.3 逾期律师函审批通过

1. 生成催款通知书或租金逾期处理通知书。
2. 合同/账单标记为法务处理中。
3. 保存律师函审批文件。
4. 后续仍未处理时，可继续进入退租/解除合同流程。

### 13.4 退租审批通过

1. 合同状态变为退租审批通过/待交接。
2. 生成退租审批表。
3. 生成合同解除补充协议。
4. 合同列表/详情出现“发起房屋验收”。

### 13.5 房屋验收通过

1. 生成或归档房屋退租交接验收单。
2. 上传费用支付凭证、现场照片和验收附件。
3. 计算应退押金、扣款和未结清费用。
4. 验收通过后自动生成押金退还付款单，由业务人员继续发起付款审批。
5. 验收通过时合同状态变为已终止并释放房源；押金退还完成后回写付款单状态。

1. 先在模式设计里重新部署合同审批，确保当前激活的 `contract_sign` 挂载表单 key `contract`，不要继续使用旧的 `wf_ex_contract`。
2. 确认五个合同相关流程都能从“新建流程”正常打开内置表单：`contract_sign`、`pay`、`roomreview`、`termination`、`law`。
3. 确认合同创建页能拿到已入驻企业、房源、合同模板、租期、租金、押金这些基础字段。
4. 确认盖章合同上传后是否立即生成付款计划；明天第一批开发建议按“上传盖章合同成功即生成账单”处理。
5. 确认打印模板优先级：第一批先跑通合同会签审批表、合同正文、付款通知单。
6. 确认所有流程回写都走统一 `biz_contract_workflow_record`，不要把多次退租、付款、逾期流程挤进合同主表单字段里。

## 17. 风险点

1. 旧模板里有 `.doc`、`.xls`，后端填充和 PDF 转换前最好先统一转换为新格式。
2. “打印一模一样”不能靠页面 HTML，需要模板引擎和 PDF 转换链路稳定。
3. 同一个合同可能发起多次付款、逾期、退租流程，必须用流程记录表承载多实例。
4. 流程 key 由用户配置，业务代码不能只认一个固定 key。
5. 后续上传的新模板不能自动知道业务字段含义，至少需要模板字段配置；自动识别只能作为增强能力。

## 18. 推荐落地顺序

最稳的推进方式：

1. 先确认五个合同相关流程在部署管理里的流程名称、流程 key、内置表单 key。
2. 先做合同审批一条线，因为它是后面账单、付款、退租的起点。
3. 再做模板打印底座，不然后面每个流程都会重复造打印逻辑。
4. 再做付款流程，因为付款账单已经由合同创建生成。
5. 再做逾期流程，因为它依赖付款计划。
6. 最后做退租流程，因为它依赖合同、付款、房源释放和押金退还。
