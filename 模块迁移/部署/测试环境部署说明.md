# 吴中金控企业服务平台测试环境部署说明

本文档用于部署一套测试环境，覆盖后端 BladeX-Boot、前端 Saber3、MySQL、Redis、MinIO、Nginx 与工作流插件。测试环境建议先按单机部署落地，后续再根据访问量拆分数据库、文件服务和应用服务。

## 1. 部署目标

| 模块 | 项目目录 | 测试环境建议 |
| --- | --- | --- |
| 后端服务 | `Boot` | Java Jar 方式启动，端口 `8080` |
| 前端管理端 | `saber3` | Vite 打包后由 Nginx 托管，端口 `80` 或 `2888` |
| 数据库 | MySQL | 测试库名 `wzjk`，本地开发库为 `bladex_boot` |
| 缓存 | Redis | 单机 Redis，必须配置密码 |
| 文件服务 | MinIO / OSS | 用于合同、审批表、附件、导出文件 |
| 工作流 | BladeX Workflow / Flowable | 使用协同办公插件，流程在部署管理中激活 |

## 2. 服务器准备

建议测试服务器配置：

| 资源 | 建议 |

## 3. 数据库初始化

### 3.1 创建数据库

正式整理后的测试环境建议使用库名 `wzjk`，本地开发库仍可继续使用 `bladex_boot`。如果是全新环境，先创建目标库：

```sql
CREATE DATABASE IF NOT EXISTS wzjk
  DEFAULT CHARACTER SET utf8mb4
  COLLATE utf8mb4_general_ci;
```

如果仍按 BladeX 默认库名部署，也可以把下面命令中的 `wzjk` 替换为 `bladex_boot`。

### 3.2 当前推荐方式：从本地完整库同步

当前阶段，最简单、最不容易漏菜单和业务表的方式，是把本地开发库 `bladex_boot` 完整同步到测试库 `wzjk`。本次 `192.168.19.16` 测试环境就是按这个方式处理的。

这样做会把本地的菜单、角色菜单、业务表、字典、流程表、合同/园区/财务/企业服务等数据一起搬到测试环境。适合测试库刚搭好、数据不重要、页面和菜单需要与本地保持一致的情况。

不要把旧库名 `ry_ics` 直接替换成 `wzjk` 后执行。`ry_ics` 只是历史数据迁移来源库，不是测试环境运行库。

### 3.3 操作步骤

当本地 `bladex_boot` 已经是完整可用状态，而服务器 `wzjk` 是空库、缺表或菜单不完整时，按下面步骤同步。

本地导出：

```bash
mysqldump --protocol=TCP -h localhost -P 3306 -u root -proot \
  --single-transaction --routines --triggers --events \
  --default-character-set=utf8mb4 --add-drop-table \
  bladex_boot > /private/tmp/bladex_boot_full_for_wzjk.sql
```

上传到服务器：

```bash
ssh root@192.168.19.16 'mkdir -p /data/bladex-park/db-backup'
scp /private/tmp/bladex_boot_full_for_wzjk.sql \
  root@192.168.19.16:/data/bladex-park/db-backup/bladex_boot_full_for_wzjk.sql
```

覆盖导入服务器 `wzjk`：

```bash
ssh root@192.168.19.16

docker exec mysql mysql -uroot -pRoot@260310 -e "
DROP DATABASE IF EXISTS wzjk;
CREATE DATABASE wzjk DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_general_ci;
"

docker exec -i mysql mysql -uroot -pRoot@260310 wzjk \
  < /data/bladex-park/db-backup/bladex_boot_full_for_wzjk.sql
```

导入后验证菜单和角色菜单：

```bash
docker exec mysql mysql --default-character-set=utf8mb4 -uroot -pRoot@260310 -D wzjk -e "
SELECT COUNT(*) AS menu_total, SUM(is_deleted = 0) AS menu_active FROM blade_menu;
SELECT COUNT(*) AS role_menu_total FROM blade_role_menu;
SELECT id, name, path FROM blade_menu
WHERE is_deleted = 0 AND parent_id = 0
  AND path IN ('/home', '/park', '/contract', '/settlement', '/enterprise', '/finance')
ORDER BY sort, id;
"
```

### 3.4 如果后端启动失败：修复 Flowable 表名

本地 macOS MySQL 默认对表名大小写不敏感，导出的 Flowable 表可能是小写 `act_*`、`flw_*`。Linux MySQL 通常区分大小写，而后端 Flowable 会查询大写 `ACT_*`、`FLW_*`。如果后端启动时报下面错误：

```text
Table 'wzjk.ACT_GE_PROPERTY' doesn't exist
Table 'wzjk.FLW_CHANNEL_DEFINITION' doesn't exist
```

说明需要把服务器库里的 Flowable 表名改成大写。执行：

```bash
# 修复 act_* -> ACT_*
docker exec mysql mysql -uroot -pRoot@260310 -N -D wzjk -e "
SELECT CONCAT('RENAME TABLE ', table_name, ' TO ', UPPER(table_name), ';')
FROM information_schema.tables
WHERE table_schema = DATABASE()
  AND BINARY table_name LIKE 'act\\_%'
ORDER BY table_name;
" > /tmp/rename_flowable_act_upper.sql

docker exec -i mysql mysql -uroot -pRoot@260310 -D wzjk \
  < /tmp/rename_flowable_act_upper.sql

# 修复 flw_* -> FLW_*
docker exec mysql mysql -uroot -pRoot@260310 -N -D wzjk -e "
SELECT CONCAT('RENAME TABLE ', table_name, ' TO ', UPPER(table_name), ';')
FROM information_schema.tables
WHERE table_schema = DATABASE()
  AND BINARY table_name LIKE 'flw\\_%'
ORDER BY table_name;
" > /tmp/rename_flowable_flw_upper.sql

docker exec -i mysql mysql -uroot -pRoot@260310 -D wzjk \
  < /tmp/rename_flowable_flw_upper.sql
```

验证：

```bash
docker exec mysql mysql -uroot -pRoot@260310 -N -D wzjk -e "
SELECT COUNT(*) FROM information_schema.tables WHERE table_schema = DATABASE() AND BINARY table_name LIKE 'ACT\\_%';
SELECT COUNT(*) FROM information_schema.tables WHERE table_schema = DATABASE() AND BINARY table_name LIKE 'act\\_%';
SELECT COUNT(*) FROM information_schema.tables WHERE table_schema = DATABASE() AND BINARY table_name LIKE 'FLW\\_%';
SELECT COUNT(*) FROM information_schema.tables WHERE table_schema = DATABASE() AND BINARY table_name LIKE 'flw\\_%';
"
```

正常结果应为大写表有数量，小写表为 `0`。本次实际修复后为 `ACT_* = 76`、`act_* = 0`、`FLW_* = 8`、`flw_* = 0`。

### 3.5 后续更新前备份

后续更新部署前，先备份数据库：

```bash
docker exec mysql mysqldump -uroot -pRoot@260310 \
  --single-transaction --routines --triggers --events \
  wzjk > /data/bladex-park/db-backup/wzjk_$(date +%Y%m%d_%H%M%S).sql
```

如果只是菜单或字段增量更新，不要每次覆盖整库，只执行本次新增或修改的 SQL。

## 4. 后端部署

### 4.1 修改测试环境配置

后端测试配置文件：

```text
Boot/src/main/resources/application-test.yml
```

需要确认以下配置：

```yaml
spring:
  datasource:
    url: jdbc:mysql://测试库IP:3306/wzjk?useSSL=false&useUnicode=true&characterEncoding=utf-8&serverTimezone=GMT%2B8&allowPublicKeyRetrieval=true
    username: 测试库账号
    password: 测试库密码
  data:
    redis:
      host: Redis地址
      port: 6379
      password: Redis密码
      database: 0

social:
  domain: http://测试前端域名或IP

oss:
  enabled: true
  name: minio
  endpoint: http://MinIO地址:9000
  access-key: MinIO账号
  secret-key: MinIO密码
  bucket-name: bladex
```

基础端口在 `Boot/src/main/resources/application.yml` 中默认为：

```yaml
server:
  port: 8080
```

当前项目 Flowable 配置为：

```yaml
flowable:
  check-process-definitions: false
  database-schema-update: false
```

因此首次部署要先保证目标库已经具备 BladeX 基础表、Flowable 表和当前业务表。当前项目不再维护 `Boot/doc/sql/business/bladex_park_business.mysql.all.sql` 作为可靠部署入口；已有测试库更新时，只执行本次新增或修改的单个增量 SQL。

### 4.2 打包后端

```bash
cd /data/bladex-park/source/Boot
mvn clean package -Dmaven.test.skip=true
```

当前 `pom.xml` 的最终 jar 名称为：

```text
target/blade-api.jar
```

复制到部署目录：

```bash
cp target/blade-api.jar /data/bladex-park/backend/blade-api.jar
```

### 4.3 启动后端

简单启动方式：

```bash
cd /data/bladex-park/backend
nohup java -jar blade-api.jar \
  --spring.profiles.active=test \
  --server.port=8080 \
  > /data/bladex-park/logs/blade-api.log 2>&1 &
```

查看日志：

```bash
tail -f /data/bladex-park/logs/blade-api.log
```

检查接口文档：

```text
http://测试服务器IP:8080/doc.html
```

### 4.4 systemd 启动方式

可选，测试环境也建议使用 systemd 管理，方便重启：

```ini
[Unit]
Description=BladeX Park API
After=network.target

[Service]
Type=simple
WorkingDirectory=/data/bladex-park/backend
ExecStart=/usr/bin/java -jar /data/bladex-park/backend/blade-api.jar --spring.profiles.active=test --server.port=8080
Restart=always
RestartSec=5
StandardOutput=append:/data/bladex-park/logs/blade-api.log
StandardError=append:/data/bladex-park/logs/blade-api-error.log

[Install]
WantedBy=multi-user.target
```

保存为：

```text
/etc/systemd/system/bladex-park-api.service
```

启动：

```bash
systemctl daemon-reload
systemctl enable bladex-park-api
systemctl start bladex-park-api
systemctl status bladex-park-api
```

## 5. 前端部署

### 5.1 确认前端环境变量

生产环境配置：

```text
saber3/.env.production
```

当前接口前缀为：

```env
VITE_APP_API=/api
VITE_APP_BASE=/
```

这表示前端所有接口会请求 `/api`，需要由 Nginx 反向代理到后端 `8080`。

如测试环境部署在二级路径，例如 `/park/`，需要同步调整：

```env
VITE_APP_BASE=/park/
```

一般测试环境建议先使用根路径 `/`，减少路由问题。


## 6. 前端访问方式

当前 `192.168.19.16` 测试环境访问地址是：

```text
http://192.168.19.16:2888
```

这里确实用到了 Nginx，但用的是宿主机上的**独立 Nginx 实例**，只监听 `2888`，不占用 `80`，也不影响已有的 `zz-server`。如果只看后端接口，可以直接访问容器映射的 `8080`；如果要访问前端页面，就需要这个独立 Nginx 来托管前端静态文件，并把 `/api` 转发到后端 `8080`。

本项目不要执行默认的：

```bash
nginx -s reload
```

因为服务器上还有其他服务。需要重载本项目前端时，只执行独立实例命令：

```bash
NGX=/data/bladex-park/bladex-park-test/nginx-standalone
nginx -p "$NGX/" -c conf/nginx.conf -t
nginx -p "$NGX/" -c conf/nginx.conf -s reload
```

实际配置和启动命令见第 14.3 节。

说明：BladeX 严格请求头开启时，导出、下载、预览接口不要使用裸 `window.open` 后端地址，前端应走带 token 的 blob 请求或通过已配置的 `/api` 代理访问。

## 7. 工作流和表单配置

当前业务审批统一走协同办公工作流插件：

| 业务 | 流程位置 | 表单方式 |
| --- | --- | --- |
| 入驻审批 | 协同办公 / 模式设计、部署管理 | 当前已改为内置表单 `entry` |
| 合同审批 | 协同办公 / 模式设计、部署管理 | 内置表单，打印时按真实模板生成 |
| 退租审批 | 协同办公 / 模式设计、部署管理 | 内置表单，审批通过后进入退租闭环 |
| 付款审批 | 协同办公 / 模式设计、部署管理 | 内置表单，审批通过后回写账单 |
| 逾期律师函审批 | 协同办公 / 模式设计、部署管理 | 内置表单，审批后生成通知/律师函 |

部署后需要检查：

1. 协同办公菜单可访问。
2. 模式设计中流程 key 与业务配置一致。
3. 部署管理中流程为激活状态。
4. 内置表单已绑定到对应流程。
5. 待办、已办、抄送、我的请求均可正常查询。
6. 流程回调通知能进入业务分发服务，合同、退租、付款、逾期状态能回写。

## 10. 后续更新部署流程

### 10.1 开发与测试环境分工

建议继续在本地开发，不建议直接在测试服务器上开发代码。推荐分工：

| 环境 | 用途 | 数据处理方式 |
| --- | --- | --- |

以后本地继续开发是正常方式。只有改了数据库结构、菜单、字典、流程配置、模板配置时，才需要同步 SQL 或再次同步数据库。普通前端 UI、后端 Java 逻辑改动，只需要重新打包部署前后端。

### 10.5 以后还会不会遇到 Flowable 表名问题

会不会遇到，取决于数据库同步方式：

| 场景 | 是否会遇到 | 原因 |
| --- | --- | --- |
| 只重新打包前端、后端，不动数据库 | 不会 | 表结构没有变化。 |
| 在服务器直接执行 `Boot/doc/sql/flowable/flowable.mysql.all.create.sql` 初始化 | 通常不会 | 脚本本身创建的是后端期望的表名。 |
| 从本地 macOS MySQL `mysqldump` 后导入 Linux MySQL | 可能会 | 本地表名可能被保存为小写，Linux 区分大小写。 |
| 后续整理出 deploy 专用 SQL，并在 Linux 上直接初始化 | 不会或概率很低 | 不再依赖本地 macOS 的表名状态。 |

因此，日常开发不需要改成在测试环境开发。继续本地开发、打包部署即可；只有整库从本地搬到 Linux 服务器时，才需要执行 3.4 的表名修复检查。

## 12. 常见问题

### 12.1 后端启动失败

先看完整错误，不要只看 Maven 最后的 `BUILD FAILURE`：

```bash
tail -n 300 /data/bladex-park/logs/blade-api.log
```

重点检查：

- MySQL 地址、账号、密码是否正确。
- Redis 地址、密码是否正确。
- Flowable 表是否已经导入。
- 业务 SQL 是否遗漏字段。
- 端口 `8080` 是否被占用。

### 12.2 SQL 重复字段

例如：

```text
Duplicate column name 'xxx'
```

说明该脚本可能已经执行过。处理方式：

1. 查询字段是否存在。
2. 若字段存在，跳过该 SQL 或仅执行后续未执行部分。
3. 不要直接删字段，避免影响已有测试数据。

### 12.3 前端刷新 404

Nginx 必须配置：

```nginx
try_files $uri $uri/ /index.html;
```

这是 Vue Router history 模式必须项。

### 12.4 接口 401 或下载失败

检查：

- 前端 `VITE_APP_API=/api` 是否正确。
- Nginx `/api` 是否正确转发到后端。
- 请求头是否携带 `Blade-Auth`。
- 严格模式下是否携带 `Blade-Requested-With: BladeHttpRequest`。
- 导出下载是否走带 token 的 blob 请求。

### 12.5 文件上传失败

检查：

- MinIO 是否启动。
- bucket `bladex` 是否存在。
- 后端 `oss.endpoint` 是否能从后端服务器访问。
- access-key / secret-key 是否正确。
- Nginx `client_max_body_size` 是否足够。
