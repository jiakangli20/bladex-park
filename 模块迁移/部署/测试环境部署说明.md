# 吴中金控企业服务平台测试环境部署说明

本文档用于部署一套测试环境，覆盖后端 BladeX-Boot、前端 Saber3、MySQL、Redis、MinIO、Nginx 与工作流插件。测试环境建议先按单机部署落地，后续再根据访问量拆分数据库、文件服务和应用服务。

## 1. 部署目标

| 模块 | 项目目录 | 测试环境建议 |
| --- | --- | --- |
| 后端服务 | `Boot` | Java Jar 方式启动，端口 `8080` |
| 前端管理端 | `saber3` | Vite 打包后由 Nginx 托管，端口 `80` 或 `2888` |
| 数据库 | MySQL | 数据库名建议 `bladex_boot` |
| 缓存 | Redis | 单机 Redis，必须配置密码 |
| 文件服务 | MinIO / OSS | 用于合同、审批表、附件、导出文件 |
| 工作流 | BladeX Workflow / Flowable | 使用协同办公插件，流程在部署管理中激活 |

## 2. 服务器准备

建议测试服务器配置：

| 资源 | 建议 |
| --- | --- |
| CPU | 4 核及以上 |
| 内存 | 8 GB 及以上 |
| 磁盘 | 100 GB 及以上，建议预留文件上传目录 |
| 系统 | Linux，CentOS / Rocky / Ubuntu 均可 |

基础软件：

```bash
java -version      # JDK 17
mvn -v             # Maven 3.8+
node -v            # Node.js 18+
npm -v             # npm，或使用 pnpm/yarn
mysql --version    # MySQL 5.7+ / 8.0
redis-server -v
nginx -v
```

建议目录：

```bash
/data/bladex-park/
├── backend/        # 后端 jar、启动脚本
├── frontend/       # 前端 dist
├── logs/           # 后端日志
├── backup/         # jar、dist、数据库备份
└── uploadPath/     # 本地上传文件目录，如不用 MinIO
```

## 3. 数据库初始化

### 3.1 创建数据库

```sql
CREATE DATABASE IF NOT EXISTS bladex_boot
  DEFAULT CHARACTER SET utf8mb4
  COLLATE utf8mb4_general_ci;
```

### 3.2 首次部署 SQL 顺序

测试环境首次建库不需要逐个执行二开业务脚本。当前已提供业务全量初始化脚本：

```text
Boot/doc/sql/business/bladex_park_business.mysql.all.sql
```

首次部署只需要按下面顺序执行 3 类 SQL：

1. BladeX 基础库。
2. Flowable 工作流基础库。
3. 吴中金控业务全量库。

注意：业务全量脚本内部会按顺序执行协同办公插件、园区管理、入驻管理、合同管理、财务管理、企业服务等 SQL。它只建议新测试库首次执行一次；已有库更新时继续走单独增量脚本。

```bash
cd /data/bladex-park/source

# 1. BladeX 基础库
mysql -h127.0.0.1 -P3306 -uroot -p bladex_boot < Boot/doc/sql/bladex/bladex.mysql.all.create.sql

# 2. Flowable 工作流基础表
mysql -h127.0.0.1 -P3306 -uroot -p bladex_boot < Boot/doc/sql/flowable/flowable.mysql.all.create.sql

# 3. 吴中金控业务全量初始化
mysql -h127.0.0.1 -P3306 -uroot -p bladex_boot < Boot/doc/sql/business/bladex_park_business.mysql.all.sql
```

如果执行业务全量脚本时报某个字段或菜单重复，通常表示该库不是全新库，或之前已经执行过部分业务 SQL。此时不要反复执行全量脚本，应先备份数据库，再按具体报错改用对应增量脚本处理。

后续更新部署前，先备份数据库：

```bash
mysqldump -h127.0.0.1 -P3306 -uroot -p bladex_boot > /data/bladex-park/backup/bladex_boot_$(date +%Y%m%d_%H%M%S).sql
```

## 4. 后端部署

### 4.1 修改测试环境配置

后端测试配置文件：

```text
Boot/src/main/resources/application-test.yml
```

需要确认以下配置：

```yaml
spring:
  datasource:
    url: jdbc:mysql://测试库IP:3306/bladex_boot?useSSL=false&useUnicode=true&characterEncoding=utf-8&serverTimezone=GMT%2B8&allowPublicKeyRetrieval=true
    username: 测试库账号
    password: 测试库密码
  data:
    redis:
      host: Redis地址
      port: 6379
      password: Redis密码
      database: 0

social:
  domain: http://测试前端域名或IP

oss:
  enabled: true
  name: minio
  endpoint: http://MinIO地址:9000
  access-key: MinIO账号
  secret-key: MinIO密码
  bucket-name: bladex
```

基础端口在 `Boot/src/main/resources/application.yml` 中默认为：

```yaml
server:
  port: 8080
```

当前项目 Flowable 配置为：

```yaml
flowable:
  check-process-definitions: false
  database-schema-update: false
```

因此必须先导入 Flowable 基础 SQL 和业务全量 SQL，再启动后端。

### 4.2 打包后端

```bash
cd /data/bladex-park/source/Boot
mvn clean package -Dmaven.test.skip=true
```

当前 `pom.xml` 的最终 jar 名称为：

```text
target/blade-api.jar
```

复制到部署目录：

```bash
cp target/blade-api.jar /data/bladex-park/backend/blade-api.jar
```

### 4.3 启动后端

简单启动方式：

```bash
cd /data/bladex-park/backend
nohup java -jar blade-api.jar \
  --spring.profiles.active=test \
  --server.port=8080 \
  > /data/bladex-park/logs/blade-api.log 2>&1 &
```

查看日志：

```bash
tail -f /data/bladex-park/logs/blade-api.log
```

检查接口文档：

```text
http://测试服务器IP:8080/doc.html
```

### 4.4 systemd 启动方式

可选，测试环境也建议使用 systemd 管理，方便重启：

```ini
[Unit]
Description=BladeX Park API
After=network.target

[Service]
Type=simple
WorkingDirectory=/data/bladex-park/backend
ExecStart=/usr/bin/java -jar /data/bladex-park/backend/blade-api.jar --spring.profiles.active=test --server.port=8080
Restart=always
RestartSec=5
StandardOutput=append:/data/bladex-park/logs/blade-api.log
StandardError=append:/data/bladex-park/logs/blade-api-error.log

[Install]
WantedBy=multi-user.target
```

保存为：

```text
/etc/systemd/system/bladex-park-api.service
```

启动：

```bash
systemctl daemon-reload
systemctl enable bladex-park-api
systemctl start bladex-park-api
systemctl status bladex-park-api
```

## 5. 前端部署

### 5.1 确认前端环境变量

生产环境配置：

```text
saber3/.env.production
```

当前接口前缀为：

```env
VITE_APP_API=/api
VITE_APP_BASE=/
```

这表示前端所有接口会请求 `/api`，需要由 Nginx 反向代理到后端 `8080`。

如测试环境部署在二级路径，例如 `/park/`，需要同步调整：

```env
VITE_APP_BASE=/park/
```

一般测试环境建议先使用根路径 `/`，减少路由问题。

### 5.2 打包前端

```bash
cd /data/bladex-park/source/saber3
npm install
npm run build:prod
```

打包产物：

```text
saber3/dist
```

复制到 Nginx 目录：

```bash
rm -rf /data/bladex-park/frontend/dist
cp -R dist /data/bladex-park/frontend/dist
```

## 6. Nginx 配置

示例配置如下，端口可按测试服务器实际情况调整为 `80` 或 `2888`。

```nginx
server {
    listen 2888;
    server_name _;

    root /data/bladex-park/frontend/dist;
    index index.html;

    client_max_body_size 100m;

    location / {
        try_files $uri $uri/ /index.html;
    }

    location ^~ /api/ {
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Blade-Requested-With BladeHttpRequest;

        rewrite ^/api/(.*)$ /$1 break;
        proxy_pass http://127.0.0.1:8080;
    }

    location ~* \.(js|css|png|jpg|jpeg|gif|svg|ico|woff2?)$ {
        expires 7d;
        access_log off;
        try_files $uri =404;
    }
}
```

重新加载：

```bash
nginx -t
nginx -s reload
```

访问：

```text
http://测试服务器IP:2888
```

说明：BladeX 严格请求头开启时，导出、下载、预览接口不要使用裸 `window.open` 后端地址，前端应走带 token 的 blob 请求或通过已配置的 `/api` 代理访问。

## 7. 工作流和表单配置

当前业务审批统一走协同办公工作流插件：

| 业务 | 流程位置 | 表单方式 |
| --- | --- | --- |
| 入驻审批 | 协同办公 / 模式设计、部署管理 | 当前已改为内置表单 `entry` |
| 合同审批 | 协同办公 / 模式设计、部署管理 | 内置表单，打印时按真实模板生成 |
| 退租审批 | 协同办公 / 模式设计、部署管理 | 内置表单，审批通过后进入退租闭环 |
| 付款审批 | 协同办公 / 模式设计、部署管理 | 内置表单，审批通过后回写账单 |
| 逾期律师函审批 | 协同办公 / 模式设计、部署管理 | 内置表单，审批后生成通知/律师函 |

部署后需要检查：

1. 协同办公菜单可访问。
2. 模式设计中流程 key 与业务配置一致。
3. 部署管理中流程为激活状态。
4. 内置表单已绑定到对应流程。
5. 待办、已办、抄送、我的请求均可正常查询。
6. 流程回调通知能进入业务分发服务，合同、退租、付款、逾期状态能回写。

## 8. 文件模板和上传目录

当前合同、审批、通知书等原始模板位于：

```text
saber3/public/系统所需材料
```

测试环境部署时需要确保这些文件随前端一起发布，或后端生成文件时能访问对应模板。建议后续正式化时将模板统一迁移到后端资源目录或 OSS 模板目录，避免前端静态资源和后端生成逻辑耦合。

MinIO 需要提前创建 bucket：

```text
bladex
```

并确认：

1. 后端配置的 `endpoint / access-key / secret-key / bucket-name` 正确。
2. 上传附件、合同模板、审批表导出文件可以正常保存。
3. 文件预览和下载地址在测试环境可访问。

## 12. 常见问题

### 12.1 后端启动失败

先看完整错误，不要只看 Maven 最后的 `BUILD FAILURE`：

```bash
tail -n 300 /data/bladex-park/logs/blade-api.log
```

重点检查：

- MySQL 地址、账号、密码是否正确。
- Redis 地址、密码是否正确。
- Flowable 表是否已经导入。
- 业务 SQL 是否遗漏字段。
- 端口 `8080` 是否被占用。

### 12.2 SQL 重复字段

例如：

```text
Duplicate column name 'xxx'
```

说明该脚本可能已经执行过。处理方式：

1. 查询字段是否存在。
2. 若字段存在，跳过该 SQL 或仅执行后续未执行部分。
3. 不要直接删字段，避免影响已有测试数据。

### 12.3 前端刷新 404

Nginx 必须配置：

```nginx
try_files $uri $uri/ /index.html;
```

这是 Vue Router history 模式必须项。

### 12.4 接口 401 或下载失败

检查：

- 前端 `VITE_APP_API=/api` 是否正确。
- Nginx `/api` 是否正确转发到后端。
- 请求头是否携带 `Blade-Auth`。
- 严格模式下是否携带 `Blade-Requested-With: BladeHttpRequest`。
- 导出下载是否走带 token 的 blob 请求。

### 12.5 文件上传失败

检查：

- MinIO 是否启动。
- bucket `bladex` 是否存在。
- 后端 `oss.endpoint` 是否能从后端服务器访问。
- access-key / secret-key 是否正确。
- Nginx `client_max_body_size` 是否足够。
